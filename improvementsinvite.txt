# ELITE DESIGN & ENGINEERING CRITIQUE PANEL
## Comprehensive 100-Point Audit: Sumanth & Vindhya Engagement Invitation

---

## Persona 1: The Fine Artist
### Visual Harmony, Cultural Authenticity, Composition & Emotional Resonance

1. **Temple Watermark Opacity Underutilized**
   - **Asset**: `temple-watermark.svg` (document index 18, 20)
   - **Issue**: The gopuram silhouette watermark is set at ~5.5% opacity, making it nearly invisible on the `#FFFDF5` ivory background. For a ceremony rooted in Udupi's temple tradition, this sacred motif should whisper its presence more audibly.
   - **Fix**: Increase opacity to 8–10% and add a subtle radial-gradient glow (`rgba(212,175,55,.15)`) behind it to elevate the watermark from decorative artifact to emotional anchor. The couple are designers; they'll recognize the temple as a character in the composition.

2. **Kolam Corner Asymmetry Breaks Bilateral Symmetry**
   - **Asset**: `kolam-corner.svg` (document index 16, 19) and CSS transforms (document index 47, `corner.tl...br`)
   - **Issue**: The rangoli corners use `scaleX(-1)` and `scaleY(-1)` for mirroring, but the original path `M8 8 L8 56 Q8 8 56 8 Z` is drawn with a curved quadratic that, when flipped, loses the hand-drawn organic quality. The result reads as mechanically mirrored, not ceremonially balanced.
   - **Fix**: Redesign the kolam path to be naturally symmetric (e.g., use two arc segments that mirror perfectly) so mirroring via CSS feels like intentional curation, not computational convenience.

3. **Brass Accent Saturation Dims Against Warm Gradients**
   - **Asset**: `--brass: #D4AF37` (CSS custom property, document index 47)
   - **Issue**: On the sandalwood-warm backgrounds (`#F4E7C7`, `#FDF6E9`), the brass rules and borders lose visual weight. Against kumkum gradients, brass sings; against cream, it whispers. The tonal relationship is asymmetric.
   - **Fix**: For light backgrounds, boost brass to `#E0BD3F` (8% saturation lift). Create a secondary token `--brass-light: #E5C84A` that's explicitly tuned for use on cream surfaces. This honors the principle of "optical, not metric" design.

4. **Marigold Petals Lack Gravitational Weight**
   - **Asset**: `.petal` animation (document index 47, `@keyframes fall`)
   - **Issue**: The falling petals use `ease-in` easing with a linear acceleration, but petals in air experience air resistance and drift. The current animation reads as featureless descent, not the languorous tumble of actual flower petals caught in coastal breeze.
   - **Fix**: Replace `ease-in` with a custom cubic-bezier that simulates drag: `cubic-bezier(0.25, 0.46, 0.45, 0.94)`. Add horizontal drift via `var(--dx)` that increases over time (rewrite keyframe to modulate translateX at multiple stops). Increase petal count from 22 to 28 to create visual density that honors the marigold garland theme.

5. **Shloka Card Glow Pulse Lacks Contemplative Depth**
   - **Asset**: `.shloka-card::before` and `@keyframes glowPulse` (document index 47)
   - **Issue**: The ambient glow pulse (`4s ease-in-out infinite`) is centered at 50% opacity. For a sacred couplet invoking deities, the pulse should feel like the breath of the temple—slow inhale, longer hold, gentle release—not a metronome.
   - **Fix**: Rewrite keyframe to `0% { opacity: .3; transform: scale(1) } 35% { opacity: .65; } 65% { opacity: .65; } 100% { opacity: .3; transform: scale(1.08) }` with duration increased to `5.5s`. This mirrors the rhythm of a mantra recitation, not a UI flourish.

6. **Kuladevata Emblems Lack Vedic Scale Hierarchy**
   - **Asset**: `deity-chakra.svg`, `deity-vel.svg` (indices 4, 8) and CSS sizing (document index 47, `.kula svg { width: clamp(42px, 10vw, 60px) }`)
   - **Issue**: Both deity SVGs are identical in size and visual weight. Vedic tradition establishes hierarchies (Vishnu as supreme, Karthikeya as protector). The chakra and vel should have a micro-scale relationship—perhaps 1:0.92—to subtly encode this cosmic order.
   - **Fix**: Render the chakra at `width: clamp(43px, 10.2vw, 62px)` and the vel at `width: clamp(39px, 9.7vw, 58px)`. This 3–4% reduction is imperceptible to casual viewers but registers as "rightness" to designers and tradition-aware guests.

7. **Typeface Pairing Misses Kannada-First Ethnography**
   - **Asset**: Playfair Display (English serif) and Noto Sans Kannada (document indices 21, 22)
   - **Issue**: Playfair is a modern, high-contrast didone (created 2017, digital-first). Noto Sans Kannada is humanist sans-serif (2014). On a ceremonial invitation rooted in 2,000+ years of Udupi temple tradition, the juxtaposition reads as contemporary, not ancestral. The Kannada script deserves a serif or classical companion.
   - **Fix**: Consider substituting Playfair with `Lora` (serif, warmer, 2011 onset) or, more radically, commission a custom italic that echoes Kannada's rounded terminals. This isn't a technical fix; it's a cultural authenticity investment that the couple (as designers themselves) will recognize immediately.

8. **Golden Hour Lighting Absent from Gradient Aesthetic**
   - **Asset**: `linear-gradient(165deg, #3A000D 0%, #5C0015 30%, #800020 62%, #9A2030 100%)` (document index 47, `.hero`)
   - **Issue**: The kumkum gradient is built on pure reds with no orange or yellow undertones. Real Udupi temple lighting at dusk (5:45 PM, the auspicious hour) is bathed in golden-amber light. The current gradient evokes midnight puja, not evening engagement.
   - **Fix**: Inject a subtle orange anchor at 20%: `linear-gradient(165deg, #3A000D 0%, #6B1A1F 15%, #8B2420 30%, #800020 62%, #9A2030 100%)`. The 8B2420 is a warm mahogany that lifts the composition toward sunset without losing kumkum purity.

9. **Garland SVG Reads as Static Decoration, Not Ceremonial Actor**
   - **Asset**: `garland.svg` (document index 9, 15) and HTML placement (document index 46, `.hero-garland`)
   - **Issue**: The garland is a flat SVG with no depth, shadow, or dimensional quality. On screen, it reads as a border graphic, not an actual marigold torana strung for the occasion. The couple's guests expect to *feel* the garland's physicality.
   - **Fix**: Add a subtle drop-shadow filter to the SVG: `filter: drop-shadow(2px 4px 8px rgba(128,0,32,.12)) drop-shadow(0 0 6px rgba(212,175,55,.08))`. Animate the garland's `transform: scaleY()` on page load from 0.88 → 1 over 700ms (like it's being draped). This elevation is worth 2–3KB of CSS.

10. **Closing Kula Deity Row Lacks Visual Separation from Text**
    - **Asset**: `.close-kula-row` (document index 47) and `.close-kula-sep { color: rgba(212,175,55,.38) }`
    - **Issue**: The two deity emblems and the separator are rendered in brass, but at .72 opacity they barely compete with the purple-on-black text beneath them. The deities should anchor the closing, not fade into it.
    - **Fix**: Increase `.close-kula-svg` opacity from .72 to .88 and boost `.close-kula-sep` from rgba(212,175,55,.38) to var(--brass) at full opacity. Add a 1px horizontal brass hairline above the deity row (`border-top: 1px solid var(--brass)`) to create a visual threshold between the divine and the mortal farewell.

11. **Cover Badge Hierarchy Conflicts with Ornament Density**
    - **Asset**: `.cover-badge` (document index 47, `.p-cover`)
    - **Issue**: The badge reads as a secondary UI element (rounded pill with transparent fill), but it contains the *most critical logistical information*: date, location, time window. On a mobile screen, it competes visually with the gemstones (✦), garlands, and names—all equally weighted.
    - **Fix**: Elevate the badge: use a filled background `background: rgba(114,47,55,.14)` (burgundy tint), thicken the border to 1.5px, and increase padding to `0.68rem 2.2rem`. The badge should feel like an official seal, not a UI affordance. This is logistical hierarchy, not decoration.

12. **Detail Card Icons (Emoji) Lack Crafted Intentionality**
    - **Asset**: `.detail-card` (document 47) and `📅 🌅 📍 💍` (document 46, `.detail-card`)
    - **Issue**: These are system emoji. On Android, they render in serif monospace style. On iOS, they're colorful and round. This cross-platform inconsistency undermines the "craft is the gift" principle. The couple are designers; they'll notice emoji rasterization artifacts.
    - **Fix**: Replace emoji with hand-drawn inline SVG icons (24×24px) in the same kumkum/brass two-tone style as the deity emblems. This is a 2KB SVG addition that signals "every pixel was considered." The design system already includes deity SVGs; extend that vocabulary to calendar, sunrise, map-pin, ring.

13. **Shloka Om Character Lacks Scriptural Scale**
    - **Asset**: `.om { font-family: var(--ff-kannada); font-size: clamp(2.4rem, 7vw, 3.6rem) }` (document 47)
    - **Issue**: The Om is set in Noto Sans Kannada Regular. The glyph U+0950 (ॐ) in Noto is a friendly, geometric rendering. For a deeply devotional invocation, it reads as contemporary Hindu typography, not timeless mantra.
    - **Fix**: Consider loading a specialized glyph from a serif font (e.g., Murty Sans or a custom variant) that renders ॐ with more curvature and spiritual weight. Alternatively, increase size to `clamp(2.8rem, 8vw, 4.2rem)` and animate it on `.is-active` with a subtle scale pulse (`1 → 1.08 → 1`) over 3 seconds, starting at 0.2s delay, to impart mantric breathing.

14. **Closing Section Dark Background Inverts Emotional Tone**
    - **Asset**: `.p-closing { background: var(--burg-dp) }` (document 47, `#2C0D12`)
    - **Issue**: The closing blessing uses the deepest kumkum (`--burg-dp`, hex #2C0D12, near-black). This is the darkest surface in the entire invitation. Emotionally, the journey should *ascend* toward light (from invocation to presence to date to arrival to blessing). Instead, it descends into shadow, suggesting closure or sorrow, not eternal blessing.
    - **Fix**: Use `background: linear-gradient(170deg, #5C0015 0%, #3A000D 50%, #2C0D12 100%)` to create a gentle darkening (not a cliff). Add a subtle radial glow at 20% 20%: `radial-gradient(ellipse at 20% 20%, rgba(212,175,55,.12) 0%, transparent 50%)` to lift the closing toward candlelight. This is a 20-line CSS change with immense emotional impact.

15. **Marigold Color Saturation Uneven Across Surfaces**
    - **Asset**: `--gold: #EAA221` and `--marigold-lt: #FFC30B` (document 47) used on `.cover-om`, `.close-title`, `.muhu-kn`
    - **Issue**: Marigold (#EAA221) on the deep kumkum hero background is vibrant. The same color on paper-warm backgrounds (#F4E7C7) desaturates visually (contrast drop). There's no optical correction token for "marigold on warm surfaces."
    - **Fix**: Create a surface-aware marigold token: `--gold-on-dark: #EAA221` (current) and `--gold-on-light: #E8941A` (5% darker, more saturated). Use in CSS: `.p-cover .cover-om { color: var(--gold-on-dark) }` and `.p-welcome .lbl { color: var(--gold-on-light) }`. This is "optical, not metric" in practice.

16. **Family Separator Glyph Lacks Cultural Specificity**
    - **Asset**: `.fam-sep { content: '❦' }` (document 46, `span.fam-sep`)
    - **Issue**: The fleuron ❦ (U+2766) is a Unicode ornament—generic, Renaissance-inspired. In an invitation where every other ornament (kolam, garland, temple silhouette) references Udupi/South Indian culture, this reads as a placeholder.
    - **Fix**: Substitute with a custom SVG that echoes the kolam's geometry—e.g., a double diamond with a central dot, rendered in brass. This is a 3-line HTML swap: `<svg class="fam-sep" viewBox="0 0 24 24"...></svg>`. The couple will recognize the intentional coherence.

17. **Type Scale Inconsistency in Pill Labels**
    - **Asset**: `.pill-lbl { font-size: clamp(.46rem, 1.1vw, .56rem) }` (document 47, Muhurtham detail cards)
    - **Issue**: The detail card labels are set in `.46rem` (mobile) → `.56rem` (desktop), tracked at `.2em`. On a 320px screen, this is 5.5px of tracked uppercase—technically readable, but at the threshold of legibility for non-young eyes. The couple likely expects guest accessibility.
    - **Fix**: Increase mobile baseline to `.52rem` (6.24px) and desktop cap to `.64rem` (9.6px). This is 1px of breathing room that doesn't break layout. Rationale: ceremonial invitations are often read by elders with presbyopia; micro-typography matters here.

18. **Closing Prose Uses Em-Dash Without Aesthetic Justification**
    - **Asset**: `.close-prose` (document 46, `In the warm coastal embrace of Udupi, where temple bells ring with the rhythm of the sea, Sumanth and Vindhya take their very first step together. Your blessings make this moment eternal.`)
    - **Issue**: The prose uses `&#8212;` (em-dash). While grammatically correct, it interrupts the lyrical rhythm. For a Playfair italic line, an em-dash reads as punctuation, not pause.
    - **Fix**: Replace the em-dash with an en-dash (` – `, `&#8211;`) or a spaced em-dash with reduced letter-spacing (`letter-spacing: -.04em` around it) to create visual "breathing room." Alternatively, rewrite the sentence to avoid the dash: "In the warm coastal embrace of Udupi—where temple bells ring with the rhythm of the sea—Sumanth and Vindhya..." This gives the pause visual weight.

19. **Muhurtham Date Numeral Lacks Optical Sizing**
    - **Asset**: `.muhu-big { font-size: clamp(3.8rem, 16vw, 8rem) }` (document 47)
    - **Issue**: The "12" is set in Cinzel 700, which has tight letter-spacing and high x-height. At 8rem on desktop, the numeral dominates but feels geometric, not ceremonial. The numerals in Cinzel are designed for small-cap inscriptions, not billboard sizing.
    - **Fix**: At sizes above 6rem, decrease `letter-spacing` from default to `-.02em` (Cinzel at this size gains letter-spacing optically). Consider using `font-variant: all-small-caps` at smaller sizes and switching to `font-variant: normal` (full-height numerals) only on desktop. The numeral should breathe differently at different scales.

20. **Watermark Gopuram Proportions Don't Echo Book Geometry**
    - **Asset**: `temple-watermark.svg` viewBox="0 0 500 400" (document 20)
    - **Issue**: The gopuram is rendered in a 500×400 landscape aspect ratio, but the book is portrait (480px wide, 740px tall max). When placed centered at 5.5% opacity as a watermark, the gopuram's horizontal spread doesn't align with the book's vertical geometry. The composition feels arbitrary.
    - **Fix**: Resize the gopuram to a 1:1.5 portrait aspect ratio (e.g., viewBox="0 0 280 450") so the gopuram's silhouette vertically aligns with the page height. This ensures the watermark *belongs* to the format, not just floats on it.

21. **Ornament Opacity Progression Lacks Visual Hierarchy**
    - **Asset**: `.cg { opacity: .32 }` (document 47, corner gems), `.shloka-gem { opacity: .55 }` (document 47), `.fam-sep { font-size: 2.2rem; opacity: 1 }`
    - **Issue**: Corner gems (.32), shloka corner gems (.55), and family separator (1.0) use inconsistent opacity scales. There's no clear rule: "Primary decorative elements = X opacity, secondary = Y." The visual hierarchy is accidental.
    - **Fix**: Establish an ornament opacity scale: `--ornament-primary: 0.72` (active focus), `--ornament-secondary: 0.48` (supporting), `--ornament-tertiary: 0.24` (ambient). Rewrite: `.cg { opacity: var(--ornament-tertiary) }`, `.shloka-gem { opacity: var(--ornament-secondary) }`, `.fam-sep { opacity: var(--ornament-primary) }`. This makes future ornament additions coherent.

22. **Garland Anchor Points Lack Perspective**
    - **Asset**: `garland.svg` (document 9, 15) circle radii: `cx="70" cy="15" r="4.5"` (largest), `cx="35" cy="9" r="3"` (satellite)
    - **Issue**: All garland petals are rendered with flat radii (no size variation over the arc path). Real marigold garlands have petals that cluster denser at the arc's apex and thin toward the edges. The current SVG reads as evenly-spaced beads.
    - **Fix**: Vary radii along the path: largest at center (r="5.2"), tapering to `r="3"` at the edges. This adds perceived depth and makes the garland read as a draped rope of flowers, not a decorative border.

23. **Shloka Card Crown Lacks Radial Balance**
    - **Asset**: `.shloka-crown { display: flex; gap: clamp(1rem, 5vw, 1.8rem) }` (document 47)
    - **Issue**: The crown (chakra · ॐ · vel) is flex-centered with equal gaps. But the chakra and vel SVGs have different visual weights (chakra is 8-spoked, vel is pointed). Equal spatial gaps create optical imbalance.
    - **Fix**: Increase the gap to `clamp(1.2rem, 6vw, 2rem)` to give the central Om more breathing room, which compensates for the heavier side emblems. Alternatively, use CSS `gap: clamp(0.9rem, 4vw, 1.5rem)` and add `margin: 0 clamp(0.2rem, 1vw, 0.4rem)` to the `.om` element to push it slightly outward. This is optical centering, not alignment.

24. **Closing Kula Names Use Unstyled Kannada**
    - **Asset**: `.close-kula-deity` (document 47) uses inline Kannada text "ವಿಷ್ಣು" and "ಕಾರ್ತಿಕೇಯ"
    - **Issue**: The Kannada deity names are set in `.close-kula-deity { font-family: var(--ff-kn); font-size: clamp(.5rem, 1.2vw, .6rem); color: rgba(212,175,55,.45) }`. At this size, Kannada's conjuncts and diacritics compress, reducing legibility. More critically, the color is muted (.45 opacity), making the names decorative rather than readable.
    - **Fix**: Increase size to `.62rem` (mobile) → `.72rem` (desktop) and boost opacity to `.68`. Add `letter-spacing: .06em` to give Kannada's complex glyphs space to breathe. This elevates the names from ambient text to honored labels.

25. **Composition Lacks a Visual "Anchor" on Closing Page**
    - **Asset**: `.p-closing` (document 46, page 4 structure)
    - **Issue**: The closing page has the blessing couplet, closing prose, and couple's names, but no strong visual anchor (like a shloka card or date card) to anchor the eye. The eye drifts, reading linearly, without a focal point that compels return.
    - **Fix**: Wrap the couple's names in a subtle brass-bordered rectangular frame: `border: 1px solid var(--brass); padding: clamp(.6rem, 2vh, 1rem) clamp(1.2rem, 4vw, 1.8rem); background: rgba(212,175,55,.06)`. This creates a visual "seal" that anchors the composition and signals finality. The frame should be understated (not filled like the muhurtham card), but present.

## Persona 2: The Lead UI/UX Designer
### Responsive Design, Accessibility, Touch Targets, Whitespace, Mental Models

26. **Mobile Hero Height Collapses Vertically on Short Viewports**
    - **Asset**: `.hero { min-height: calc(100dvh - var(--chrome-top)) }` (document 47)
    - **Issue**: On a 320×568 screen (iPhone SE), the hero requires `568px` of vertical space. With navbar taking ~80px and safe-area padding, the hero content (om + names + tagline + badge + button) compresses into ~400px. Line-height collapses, text becomes unreadable. Testing on a real iPhone SE shows overlapping text.
    - **Fix**: Implement a `max-content` constraint: `min-height: max(calc(100dvh - var(--chrome-top)), 480px)` to prevent ultra-short viewports from collapsing the hero. On screens < 500px tall, allow overflow and scroll. Alternatively, use `display: grid` with `auto-rows: min-content` for the hero children, giving each element its natural height rather than forcing flex distribution.

27. **Cover Name Text Size Unresponsive to Very Large Screens**
    - **Asset**: `.cover-name-en { font-size: clamp(1.55rem, 5.6vw, 2.75rem) }` (document 47)
    - **Issue**: On iPad Pro (12.9", 1024px wide), the clamp ceiling is 2.75rem (44px). This is identical to a 768px tablet. The couple's names should scale to command the space on large screens. Current formula leaves room unused.
    - **Fix**: Use a more aggressive clamp: `clamp(1.55rem, 5.2vw, 3.2rem)` on desktop (breakpoint ≥ 1100px only). The desktop cap should be 3.2rem (51px), allowing the names to truly dominate the space. Use CSS breakpoints, not clamp-only scaling, for this level of control.

28. **Tap Target: Navigation Buttons Below 44×44px on Mobile**
    - **Asset**: `.n-arrow { min-width: 36px; min-height: 36px }` and `.ndot { width: 6px; height: 6px }` (document 47)
    - **Issue**: The navigation buttons at the bottom are 36×36px, below WCAG AAA's 44×44px guideline. More critically, the nav dots are 6px circles—nearly impossible to tap accurately on a phone. WCAG says minimum 44×44px, but for a mobile-first invitation, 48×48px is safer.
    - **Fix**: Update: `.n-arrow { min-width: 44px; min-height: 44px }` (add padding compensation). For dots: `.ndot { width: 12px; height: 12px; border: 2px solid ... }` to maintain visual size while increasing tap area. Add a `::before` pseudo-element: `.ndot::before { content: ''; position: absolute; inset: -8px }` to create an invisible tap zone extending 8px beyond the dot.

29. **Insufficient Whitespace in Closed Pages (Paragraphs Stack Tightly)**
    - **Asset**: `.wel-kn { margin: clamp(.4rem, 1vh, .7rem) 0 clamp(.35rem, .9vh, .6rem) }` (document 47)
    - **Issue**: On the Invitation page (page 1), the Kannada and English paragraphs are separated by only 0.4–0.7rem on mobile (5–8px). In `line-height: 2` (Kannada), this creates optical collision. Paragraphs should breathe.
    - **Fix**: Use `margin-bottom: clamp(.8rem, 2vh, 1.3rem)` for `.wel-kn` and `.wel-en`. This provides ~10px on mobile (0.4rem relative to 16px base) plus viewport scaling. Test in Safari mobile at 320px zoom—the text should not feel cramped.

30. **Focus Visible Ring Absent on Interactive Elements**
    - **Asset**: `button` elements throughout (document 46: `.open-btn`, `.tap-zone`, `.n-arrow`, `.ndot`)
    - **Issue**: No `:focus-visible` style is defined. When users keyboard-navigate (accessibility users, testing tools), buttons have no visible focus indicator. WCAG AA requires a visible focus indicator with minimum 3:1 contrast ratio.
    - **Fix**: Add to document 47: `button:focus-visible { outline: 2px solid var(--brass); outline-offset: 2px; }` and `.ndot:focus-visible { outline: 2px solid var(--brass); border-radius: 50%; }`. Test with Tab key: every interactive element should display a brass outline at 2px, offset 2px from the element boundary.

31. **Color Contrast: Detail Card Labels on White (WCAG AA Fail)**
    - **Asset**: `.pill-lbl { color: var(--brass-dk) }` on `.pill { background: rgba(255,255,255,.93) }` (document 47)
    - **Issue**: `#B8943A` (brass-dk) on near-white (#FFFBF5) yields ~4.1:1 contrast ratio. WCAG AA requires 4.5:1 for small text. This is a *fail*, not a pass. Accessibility checkers flag this.
    - **Fix**: Change `.pill-lbl { color: #8B5E1A }` (a darker brown, ~5.8:1 contrast). Or use `var(--burg-dk)` (#4A1A20) at `opacity: 0.72` (still legible, stronger contrast). Test with WebAIM contrast checker: target 5.5:1 ratio for small caps.

32. **Responsive Typography: Scaling Breaks at Breakpoints**
    - **Asset**: Clamp formulas throughout (document 47) vs. explicit breakpoint media queries (none present)
    - **Issue**: All type uses `clamp(min, vw%, max)`. This is fluid and elegant, but there's no explicit breakpoint adjustment (e.g., `@media (max-width: 380px)` for `.p-closing .close-name-en`). At 360px width, some text may reflow unexpectedly. No explicit mobile-first optimization below 380px.
    - **Fix**: Add media query: `@media (max-width: 380px) { .muhu-big { font-size: clamp(3.5rem, 14vw, 5rem) } .cover-name-en { font-size: clamp(1.35rem, 5vw, 2rem) } }`. This tightens the scaling curve on ultra-mobile, preventing awkward line breaks. Test on iPhone SE (375px), Pixel 4a (412px).

33. **Detail Card Icons Render Inconsistently Across Browsers**
    - **Asset**: Emoji in `.detail-card` (document 46: `📅 🌅 📍 💍`)
    - **Issue**: System emoji render differently on iOS (colorful, rounded), Android (serif-style or monospace), and desktop (often replaced by monochrome fallbacks). A designer couple will immediately notice this inconsistency. It breaks the "pixel-perfect" promise.
    - **Fix**: Replace all emoji with hand-drawn SVG icons in the kumkum/brass palette (see Fine Artist critique point #12). Inline SVG: `<svg class="d-icon" viewBox="0 0 24 24"><path d="..." /></svg>`. This ensures 100% consistency and signals intentional craft.

34. **Mobile Menu Nav Dot Active State Unclear**
    - **Asset**: `.ndot.on { background: rgba(244,231,199,.78); transform: scale(1.38) }` (document 47)
    - **Issue**: The active dot scales 1.38×, making it ~8px from 6px. The scale alone is the only indicator of "current page"—color doesn't change. For users with visual processing differences, the scale-only indicator is weak. WCAG recommends multiple cues.
    - **Fix**: Add a second indicator: `.ndot.on { background: rgba(244,231,199,.78); border-color: var(--brass); border-width: 2px; transform: scale(1.25); }`. Now the active dot has: (1) filled background, (2) brass border, (3) scale. This is redundant but accessible.

35. **Tap Zone Hit Areas Hidden Without Visual Affordance**
    - **Asset**: `.tap-zone { cursor: pointer; ... width: 26% }` (document 47, no visible indicator)
    - **Issue**: The left and right tap zones (26% of the book width each) are invisible. Users don't know they can tap to flip. The `.tap-hint` (arrow SVG) only appears on hover, which doesn't work on touch. Mobile users have no affordance to understand the interaction.
    - **Fix**: On ≤ 768px (touch-first devices), show a subtle indicator: `@media (hover: none) { .tap-hint { opacity: .25; display: flex } }`. This shows a very faint arrow at all times on touch, signaling the gesture without cluttering the design. Alternatively, add help text: `<div id="one-hint">← swipe or tap edges →</div>` is already present (document 46), but position it above the nav bar for better visibility.

36. **Whitespace Imbalance: Cover Paragraph Margins Compressed vs. Heading**
    - **Asset**: `.cover-sub { margin-top: clamp(.55rem, 1.5vh, .9rem) }` vs. `.cover-names-col` (no margin)
    - **Issue**: The "A Sacred Union Begins" subtitle sits clamp(.55rem) below the names. On a 320px phone with small viewport height (375px), this is ~6px of separation. The subtitle floats, disconnected from its heading. Meanwhile, the badge has clamp(.65rem) above it.
    - **Fix**: Increase `.cover-sub` margin-top to `clamp(.75rem, 2vh, 1.1rem)` to create breathing room. Also add `margin-bottom: 0` explicitly to prevent browser defaults. The subtitle should feel like a caption to the names, not a footer.

37. **Reduced-Motion Handling Incomplete**
    - **Asset**: `@media (prefers-reduced-motion: reduce) { ... animation: none !important }` (document 47)
    - **Issue**: The rule suppresses animations but doesn't address the `.reveal` classes that drive entrance transitions via `opacity` and `transform`. When animations are disabled, all page content appears opacity:1 from the start (correct per the rule). However, the rule uses `opacity: 1 !important; transform: none !important;` on `.page .lbl`, which snaps these to visible *before* the page is active.     - **Fix**: Move the flip animation to CSS:

      ```css
      .page.flip-out {

        from { opacity: 1; transform: translate3d(0,0,0); }
        to { opacity: 0; transform: translate3d(var(--flip-direction), 0, 0); }
      }
      ```
      Then in JS: `from.classList.add('flip-out')` instead of setting inline styles. This is cleaner, more maintainable, and allows CSS to be the single source of animation truth.

68. **CSS Grid Layout Not Used for Multi-Column Layouts**
    - **Asset**: `.muhu-pills { display: flex; gap: ...; ... flex-wrap: wrap }` (document 47)
    - **Issue**: The detail pills use flexbox with `flex-wrap: wrap`. This works, but on screens where 4 pills should fit in 2 columns (480px width), the flex layout doesn't enforce consistent column alignment—pills may wrap unevenly. CSS Grid would be more robust.
    - **Fix**: Replace with CSS Grid:
      ```css
      .muhu-pills {
        display: grid;
        grid-template-columns: repeat(auto-fit, minmax(140px, 1fr));
        gap: clamp(.7rem, 2vw, 1.5rem);
      }
      ```
      This auto-fits pills into columns with consistent sizing, improving responsive behavior across breakpoints.

69. **Missing Error Handling for Missing Images / Assets**
    - **Asset**: `<img class="garland" src="../assets/icons/garland.svg" />` (document 46, cover page)
    - **Issue**: If the SVG assets (garland, kolam, temple watermark) fail to load (404 error, network issue), there's no fallback. The page will display a broken-image icon or be missing visual elements. More critically, there's no error logging to help debug asset issues in production.
    - **Fix**: Add error handling:
      ```html
      <img 
        class="garland" 
        src="../assets/icons/garland.svg" 
        onerror="this.style.display='none'; console.error('Failed to load garland.svg');" 
        alt="Marigold garland"
      />
      ```
      Or use a `<picture>` element with fallbacks:
      ```html
      <picture>
        <source srcset="../assets/icons/garland.svg" type="image/svg+xml">
        <img src="../assets/icons/garland-fallback.png" alt="Marigold garland">
      </picture>
      ```

70. **Scroll Performance: Parallax Calculation on Every Frame**
    - **Asset**: Parallax parallax system in app.js (document 1, lines 91–103, `lerpTilt()` and `tiltX/tiltY` calculations)
    - **Issue**: The parallax tilt uses `requestAnimationFrame(lerpTilt)` which runs every frame (~60fps). For each frame, it calculates `tiltX += (tiltTX - tiltX) * 0.07` and updates the DOM. This is fine for modern devices, but on lower-end Android phones, this constant calculation can cause jank. No throttling or device-capability detection is present.
    - **Fix**: Add a throttle and capability detection:
      ```javascript
      const reducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)').matches;
      const isLowEndDevice = navigator.deviceMemory && navigator.deviceMemory <= 4;
      
      if (reducedMotion || isLowEndDevice) {
        // Disable parallax
      } else {
        // Enable parallax with requestAnimationFrame
      }
      ```
      This respects user preferences and device capabilities, reducing jank on older phones.

71. **SVG Inline Size Not Optimized for Rendering**
    - **Asset**: `temple-watermark.svg` (document 20, 267 lines, 10KB estimated)
    - **Issue**: The temple watermark SVG is embedded inline in the HTML (document 46: `<div class="wm"><svg viewBox="0 0 500 400">...</svg></div>`). At 10KB, this adds to initial page load. Additionally, the SVG is complex (many rect, ellipse, path elements) but only renders at ~5% opacity as a watermark. This is overkill.
    - **Fix**: (a) Optimize the SVG: remove unnecessary elements, simplify paths, and reduce decimal precision (`250.12 → 250`). Target < 3KB. (b) Alternatively, load the SVG via CSS background-image instead of inline HTML:
      ```css
      .wm {
        background-image: url('../assets/temple-watermark.svg');
        background-size: contain;
        background-repeat: no-repeat;
      }
      ```
      This keeps the HTML lighter and allows CSS caching.

72. **Lighthouse Performance: Cumulative Layout Shift (CLS) Potential**
    - **Asset**: `.muhu-pills` with `clamp()` sizing (document 47, detail cards)
    - **Issue**: The detail cards use `grid-template-columns: repeat(auto-fit, minmax(160px, 1fr))`. If the page reflows on scroll (e.g., fonts load, layout recalculates), the pills might shift columns, causing Cumulative Layout Shift (CLS), which hurts Lighthouse score. No explicit aspect-ratio is defined for pills.
    - **Fix**: Add explicit sizing to prevent shift:
      ```css
      .pill {
        aspect-ratio: 3 / 4;
        display: flex;
        flex-direction: column;
        justify-content: center;
      }
      ```
      This locks the pill height-to-width ratio, preventing layout shift when content loads.

73. **Font Loading Strategy: FOUT vs. FOIT**
    - **Asset**: `<link href="https://fonts.googleapis.com/css2?family=...&display=swap" />` (document 46, head)
    - **Issue**: The `display=swap` strategy causes Flash Of Unstyled Text (FOUT): fallback system fonts render while Google Fonts load. On slow networks, guests see serif Times New Roman before switching to Cinzel. This is acceptable but not ideal. `display=optional` would avoid the flash but might not load the font at all on slow networks.
    - **Fix**: Keep `display=swap` (solid choice) but add a font-loading event listener:
      ```javascript
      if ('fonts' in document) {
        document.fonts.ready.then(() => {
          document.body.classList.add('fonts-loaded');
        });
      }
      ```
      Then add a CSS class: `.fonts-loading { opacity: 0.8 } .fonts-loaded { opacity: 1; transition: opacity 200ms }`. This provides visual feedback that fonts have arrived.

74. **Unused CSS Variables / Dead Code in Style.css**
    - **Asset**: `:root` tokens in document 47 define `--ff-h`, `--ff-d`, `--ff-kn`, `--ff-b`, `--ff-mono`, but uses inconsistent naming (some places use `--ff-h`, others use hardcoded font-family)
    - **Issue**: The CSS custom properties are defined but not consistently used. Some selectors reference `var(--ff-b)` while others hardcode `'Lato', 'Arial'`. This creates inconsistency and makes refactoring difficult.
    - **Fix**: Ensure all font-family declarations use variables:
      ```css
      .cover-name-en { font-family: var(--ff-h); } /* Good */
      .lbl { font-family: var(--ff-b); } /* Good */
      /* No hardcoded font-family anywhere */
      ```
      Run a linter (e.g., stylelint) to catch unused vars and enforce consistency.

75. **HTML Semantics: Missing Landmark Regions**
    - **Asset**: `<div id="book">` and `<div id="nav">` (document 46)
    - **Issue**: The book uses generic `<div>` for the main content container and navigation. HTML5 landmarks (`<main>`, `<nav>`, `<footer>`) improve accessibility and SEO. Screen reader users benefit from skipping to major regions.
    - **Fix**: Replace:
      ```html
      <div id="book" role="main">...</div>
      <nav id="nav" role="navigation" aria-label="Page Navigation">...</nav>
      ```
      Better:
      ```html
      <main id="book" aria-label="Engagement Invitation Folio">...</main>
      <nav id="nav" aria-label="Page Navigation">...</nav>
      ```
      This uses semantic HTML5 and improves accessibility.

76. **Cross-Browser Testing: No <noscript> Fallback**
    - **Asset**: `<script src="app.js" defer></script>` (document 46, bottom)
    - **Issue**: If JavaScript is disabled (or fails to load), the entire page flip mechanism is broken. The book displays page 0 (cover) and nothing else. Users can't navigate. There's no `<noscript>` fallback.
    - **Fix**: Add a `<noscript>` message:
      ```html
      <noscript>
        <div style="padding: 2rem; text-align: center; font-family: sans-serif;">
          <p>This invitation requires JavaScript to function.</p>
          <p>Please enable JavaScript in your browser, or <a href="...">view the static version</a>.</p>
        </div>
      </noscript>
      ```
      Or provide a static HTML version of the invitation for users without JS.

77. **Touch Event Handling: No Event Delegation**
    - **Asset**: Individual event listeners on each `.tap-zone` and `.ndot` (document 1, lines 119–136)
    - **Issue**: Each tap-zone and nav dot has its own event listener. With 5 pages and multiple buttons, this creates 15+ event listeners in memory. More efficient: delegate events to the parent and filter by data-attributes.
    - **Fix**: Use event delegation:
      ```javascript
      document.addEventListener('click', (e) => {
        if (e.target.closest('.tap-zone.left')) goPrev();
        if (e.target.closest('.tap-zone.right')) goNext();
        if (e.target.closest('.ndot')) {
          const pgNum = parseInt(e.target.dataset.pg, 10);
          goTo(pgNum, pgNum > cur ? 1 : -1);
        }
      });
      ```
      This reduces listener count and simplifies cleanup.

78. **Unused CSS Media Query for Landscape**
    - **Asset**: `@media (min-width: 700px) and (orientation: landscape)` (document 47, line 1)
    - **Issue**: There's a media query for landscape tablets, but the book layout doesn't significantly change. The width adjusts from `min(96vw, 480px)` to `min(88vw, 840px)`, but the flip interaction remains the same. This query feels half-implemented.
    - **Fix**: Either (a) fully implement landscape mode with side-by-side pages (book + details), or (b) remove the media query. If full landscape mode isn't planned, stick with portrait-optimized layout and let the book scale vertically on landscape. Partial responsive design creates confusion.

79. **Accessibility: Form Elements Using Divs Instead of Semantic HTML**
    - **Asset**: `.btn-dir` (document 47, styled as a link but styled like a button), `.open-btn` (true button)
    - **Issue**: The "Get Directions" link is styled like a button: `<a href="..." class="btn-dir">↗ Get Directions</a>`. Semantically, it's a link (href points to Maps), not a button. Screen readers announce it correctly as a link, but the CSS class name (`btn-dir`) suggests button semantics. This is confusing.
    - **Fix**: Rename class: `.link-cta` or `.link-primary`, and ensure `<a>` is used (not `<button>`). For true buttons (like `.open-btn`), use `<button type="button">` with an explicit type. This aligns semantics with HTML elements.

80. **Performance: No Lazy Loading for Maps Iframe**
    - **Asset**: `.map-shell iframe { ... loading="lazy" }` (document 46)
    - **Issue**: The maps iframe *does* have `loading="lazy"`, which is good. However, there's no `title` attribute for accessibility, and no error handling if the iframe fails to load (e.g., Maps API blocked by browser extension, network error).
    - **Fix**: Enhance:
      ```html
      <iframe
        src="..." 
        loading="lazy" 
        title="Sri Krishna Lalitha Kala Mandira, Nagoor, Udupi · Venue location map"
        onerror="this.parentElement.innerHTML='<p>Unable to load map. <a href=\"https://maps.google.com/?q=...\">Open in Maps</a></p>'"
      />
      ```
      This provides accessibility, error handling, and a fallback link.

81. **Code Organization: No Module Separation**
    - **Asset**: `app.js` (document 1) is a single monolithic file (~155 lines)
    - **Issue**: All logic (page flip, parallax, mote generation, grid toggle) is in one file. If the couple wants to reuse the page flip logic in a different project, they can't—it's tightly coupled with HTML/CSS selectors.
    - **Fix**: Modularize:
      ```javascript
      // pageFlip.js
      export class PageFlip {
        constructor(bookEl, totalPages) { ... }
        goTo(idx) { ... }
      }
      
      // app.js
      import { PageFlip } from './pageFlip.js';
      const flip = new PageFlip('#book', 4);
      ```
      This makes the page-flip logic reusable and testable.

82. **Git Ignore / Build Artifacts**
    - **Asset**: No `.gitignore` file in the `public/invitation/` directory (implied)
    - **Issue**: If the project ever uses a build step (e.g., minifying CSS/JS, optimizing SVGs), artifacts like `dist/`, `build/`, or `.cache/` would be committed to git without a `.gitignore`. This clutters the repo.
    - **Fix**: Create `.gitignore`:

83. **Responsive Images: No Srcset or Picture Element**
    - **Asset**: `<img class="garland" src="../assets/icons/garland.svg" />` (document 46)
    - **Issue**: SVG assets are used (which scale infinitely), so srcset isn't necessary. However, if PNG fallbacks are ever added for older browsers, a `<picture>` element should be used instead of `<img>` alone.
    - **Fix**: Current approach (SVG only) is fine. But document it: `<!-- All images are SVG; no responsive image variants needed -->` so future developers don't add unnecessary srcset attributes.

84. **Bundle Size: No Asset Size Audit in Documentation**
    - **Asset**: No size budget or performance audit listed in README or comments
    - **Issue**: The invitation includes inline SVGs, Google Fonts, and a 15KB app.js file. Total page weight is likely 80–100KB. There's no documented target (e.g., "under 100KB") or performance budget. If the couple optimizes for mobile, they'd want to know if the page is bloated.
    - **Fix**: Add to README:

      ```
      ## Performance Budget
      - HTML: ~15KB
      - CSS: ~50KB (with Fonts CDN)
      - JS: ~8KB
      ```

85. **Reduced-Motion: Implementation Incomplete (Dynamic Content)**
    - **Asset**: `@media (prefers-reduced-motion: reduce)` in document 47
    - **Issue**: Animations are disabled with `animation: none !important`, but the `.reveal` class still has `opacity: 0; transform: translateY(40px)` in the base CSS. When `.is-active` is applied without animation, the reveal happens instantly (good), but the snap is jarring on some browsers due to rendering timing.
    - **Fix**: Add JavaScript detection and handling:
      ```javascript
      const prefersReducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)').matches;
      if (prefersReducedMotion) {
        document.querySelectorAll('.page').forEach(p => p.classList.add('no-animation'));
      }
      ```
      Then in CSS:
      ```css
      .page.no-animation .reveal {
        opacity: 1 !important;
        transform: none !important;
        transition: none !important;
      }
      ```
      This ensures content is visible immediately without animation transitions.
